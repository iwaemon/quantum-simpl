## Per-Feature Details

### Standard Hubbard Pipeline
- **Role:** Computational physicist setting up a Hubbard model simulation
- **What they tried:** Created 10-site Hubbard model following README, ran binary, inspected output files
- **Discoverability:** README and `--help` sufficient to get started
- **Setup:** Binary worked immediately, no issues
- **Functionality:** The pipeline runs and generates all 9 expected mVMC output files. However, **the README example (`sum i=0..10` with `sites=10 pbc=true`) produces wrong results** -- the range `0..10` is inclusive on both ends (11 iterations), so with PBC the bond 0-1 gets counted twice and site 0's interaction is doubled. The correct range is `sum i=0..9`.
- **Friction points:**
  - Range semantics (`..` is inclusive, not exclusive like Rust/Python) are undocumented and surprising
  - No validation/warning when sum range exceeds site count with PBC (silent double-counting)
  - `modpara.def` uses hardcoded defaults with no way to customize
- **Doc suggestions:** Fix README example to `sum i=0..9`, explicitly document range inclusivity, add note about modpara defaults

### Heisenberg Model
- **Role:** Condensed matter physics student simulating a spin chain
- **What they tried:** Created Heisenberg chain inputs with Sp/Sm/Sz operators, ran standard pipeline
- **Discoverability:** README example is clear and intuitive
- **Setup:** No issues
- **Functionality:** **CRITICAL BUG -- the standard pipeline does not call `spin_to_fermion()` before output generation.** The tool runs without errors, prints progress messages, and writes output files -- but `trans.def` and `interall.def` contain zero data lines despite the header claiming non-zero term counts. The output functions only match `FermionCreate`/`FermionAnnihilate` patterns, silently skipping spin operator terms. This means **the entire Heisenberg model feature is broken** in the standard pipeline. The correlation pipeline (`--correlation`) does correctly convert spin operators.
- **Friction points:**
  - Silent failure is the worst kind -- tool exits 0 and writes structurally valid but empty files
  - Header/data count mismatch would also crash mVMC at runtime
  - No documentation of spin normalization convention (`Sz = (n_up - n_down)/2`)
- **Doc suggestions:** Fix the bug first, then document spin conventions and show expected output for verification

### YS Transform
- **Role:** QMC researcher using Yokoyama-Shiba transformation
- **What they tried:** Compared output with/without `--ys-transform` on a 4-site Hubbard model
- **Discoverability:** Feature is **not mentioned in the README at all**. Only discoverable via `--help` or CLAUDE.md
- **Setup:** No issues
- **Functionality:** Works correctly. The transform properly flips down-spin hopping signs, generates diagonal one-body terms, moves interactions to `coulombintra.def`, and updates `namelist.def`
- **Friction points:**
  - CLAUDE.md documents `--ys_transform` (underscore) but actual CLI flag is `--ys-transform` (hyphen) -- **documentation bug**
  - No explanation of what the transform does or why to use it
  - `offset=-0` display (cosmetic negative zero)
- **Doc suggestions:** Add YS Transform section to README, fix flag name in CLAUDE.md, explain the physics

### Correlation Pipeline
- **Role:** Physicist computing correlation functions
- **What they tried:** Spin-spin correlations (`Sz*Sz`), density-density (`n*n`), full `S.S`, single-site `Sz(i)`, mixed correlations, explicit coefficients, both pipelines together
- **Discoverability:** **No example of what a correlation input file looks like.** Had to read integration test source code to figure out the format
- **Setup:** No issues
- **Functionality:** Core feature works correctly. All correlation types produced correct output. However:
  - `namelist.def` does **not** include `OneBodyG cisajs.def` or `TwoBodyG cisajscktaltdc.def` entries when running both pipelines -- mVMC won't find the files without manual editing
  - Coefficients are only in `correlation_summary.txt`, not in the `.def` files (matches mVMC format but undocumented workflow)
- **Friction points:**
  - Zero example correlation input files in the repository
  - No documentation of the post-processing workflow (combining mVMC output with coefficients)
  - `cisajs.def` is empty for two-body correlations (correct but potentially confusing)
- **Doc suggestions:** Add correlation input examples, document post-processing workflow, add OneBodyG/TwoBodyG to namelist.def, ship example files

### Open Boundary Conditions
- **Role:** Physics student simulating a finite open chain
- **What they tried:** Compared `pbc=true` vs `pbc=false`, tested "wrong" and "right" sum ranges
- **Discoverability:** Well documented in README
- **Setup:** No issues
- **Functionality:** Works correctly. OBC produces 36 transfer terms (9 bonds) vs PBC's 40 (10 bonds). Warning message is helpful: "4 term(s) dropped due to out-of-range site indices (OBC mode)"
- **Friction points:**
  - The docs recommend `sum i=0..9` for `sites=10 pbc=false`, but `i=9` still generates out-of-range terms (`c窶(9,s) c(10,s)`). The truly clean range is `sum i=0..8`
  - No `--strict` mode to error instead of warn on dropped terms
- **Doc suggestions:** Fix recommended range to `sum i=0..8`, or clearly state that warnings are expected even with the recommended range

## Issues Found

**Ordered by severity:**

1. **CRITICAL: Heisenberg model produces empty output files.** The standard pipeline does not convert spin operators to fermion operators before output generation. `trans.def` and `interall.def` have zero data lines despite non-zero headers. This would crash mVMC. The `spin_to_fermion()` call exists in the correlation pipeline but is missing from the standard pipeline.

2. **HIGH: README example has wrong sum range.** `sum i=0..10` with `sites=10 pbc=true` causes double-counting of bond 0-1 and site 0 interaction due to inclusive range semantics. Should be `sum i=0..9`.

3. **HIGH: Range semantics undocumented.** The `..` syntax is inclusive on both ends, which is surprising for users familiar with Rust (`0..10` is exclusive) or Python (`range(0,10)` is exclusive). No documentation explains this.

4. **MEDIUM: `--ys_transform` vs `--ys-transform` mismatch.** CLAUDE.md and README use underscore, but CLI expects hyphen. Clap suggests the correct form, but docs should be fixed.

5. **MEDIUM: Correlation pipeline doesn't update namelist.def.** When both pipelines run, `namelist.def` lacks `OneBodyG`/`TwoBodyG` entries, requiring manual editing.

6. **MEDIUM: YS Transform absent from README.** A significant feature with no user-facing documentation.

7. **LOW: OBC docs recommend incorrect range.** `sum i=0..9` still triggers warnings; `sum i=0..8` is the clean range.

8. **LOW: No example input files.** The `examples/` directory is empty. Users must reverse-engineer DSL syntax from test code.

9. **LOW: `offset=-0` display.** Cosmetic negative zero in YS transform classification output.

## Suggestions

**Ordered by impact:**

1. **Fix Heisenberg bug:** Add `spin_to_fermion()` conversion to the standard pipeline (before normal ordering), or detect spin operators at output time and emit a clear error.

2. **Fix README example:** Change `sum i=0..10` to `sum i=0..9` in both README.md and README.ja.md.

3. **Document range semantics:** Add to DSL syntax section: "The range `start..end` is inclusive on both ends. For N sites with PBC, use `sum i=0..N-1`."

4. **Fix flag name in docs:** Change `--ys_transform` to `--ys-transform` in CLAUDE.md and README.

5. **Add example files:** Create `examples/hubbard.def`, `examples/heisenberg.def`, `examples/correlation.def` with correct, tested input files.

6. **Add YS Transform section to README:** Explain what it is, why to use it, and show the CLI flag.

7. **Add correlation examples to README:** Show what a correlation input file looks like and the expected output.

8. **Update namelist.def generation:** Include `OneBodyG`/`TwoBodyG` entries when correlation pipeline runs alongside standard pipeline.

9. **Add `--strict` flag:** Exit with error instead of warning when OBC drops terms.

10. **Add sum range validation:** Warn when sum range exceeds `sites-1` with PBC to prevent silent double-counting.
